# XSP Agent Issues — March 31, 2026 Post-Mortem

## Incident Summary

On March 31, 2026, the XSP trading agent experienced a cascade of failures that left it unable to trade for ~2.5 hours (12:21–14:55 ET), missing a ~67-point SPX rally (~12:25–13:04 ET). The root cause was a failed position close that corrupted internal state, combined with insufficient error handling.

---

## Issues

### 🔴 P0-1: Failed Close Still Deletes Position From Memory

**File**: `src/agent/position-manager.ts`, `checkExits()` method  
**Status**: 🔧 FIXING  

**Problem**: When a sell/close order is rejected by Tradier (e.g., qty mismatch), the position manager logs a warning but **still removes the position from its internal tracking map** (`this.positions.delete(id)`). This means:
- The agent thinks it has no open positions
- The broker still holds the position, tying up buying power
- All subsequent buy orders are rejected for insufficient buying power
- The agent has no mechanism to detect or recover from this state

**What happened**: At 12:16 ET, the sell for 2x P643 was rejected ("more shares than your current long position"). The agent removed it from memory and logged phantom P&L of -$79. The broker still held 1 contract, locking buying power for the rest of the session.

**Fix**: If `result.error` is set on a close attempt, do NOT delete the position from the internal map. Log a critical error, keep the position tracked, and retry the close on the next cycle. Add a `closeFailCount` tracker to positions so repeated failures can be escalated.

---

### 🔴 P0-2: No Circuit Breaker for Persistent Buying-Power Rejections

**File**: `agent-xsp.ts`, entry execution + main loop  
**Status**: 🔧 FIXING  

**Problem**: When buy orders are rejected for "not enough buying power," the agent retries every 30 seconds indefinitely. The existing backoff (3 rejections → 5 min pause) just delays the same failing attempt. There is no mechanism to:
1. Query the broker for ghost positions consuming capital
2. Halt trading for the session after a sustained rejection streak  
3. Refresh buying power from the broker to get an updated picture

**What happened**: From 12:21 ET to market close, **228 consecutive orders** were rejected for buying power, spamming the Tradier API uselessly.

**Fix**:
- After 3 consecutive buying-power rejections, query broker for open positions and reconcile
- After 5 consecutive buying-power rejections, halt entries for the rest of the session
- On any buying-power rejection, refresh buying power from the broker immediately
- Log a CRITICAL alert so the account-monitor can pick it up

---

### 🔴 P0-3: No Price-vs-Budget Check Before Order Submission

**File**: `agent-xsp.ts`, `executeEntry()` function  
**Status**: 🔧 FIXING  

**Problem**: The agent queries buying power once at startup and caches it. It never checks whether the option's ask price exceeds the per-trade budget before submitting an order. This leads to orders that will obviously be rejected.

**What happened**: At 9:30 AM, the agent bought `XSP260331C00635000 @ $7.41` ($741/contract) with only a $178 per-trade budget. This consumed buying power and set up the cascade.

**Fix**: Before submitting any buy order, check `ask * 100 > baseDollarsPerTrade`. If too expensive, log and skip — don't waste an API call on an order that will be rejected.

---

### 🟠 P1-4: SPX Price = 0.00 Not Guarded

**File**: `agent-xsp.ts`, `runCycle()` function  
**Status**: 🔧 FIXING  

**Problem**: Three times today, the data service returned SPX price as `0.00`. The agent continued cycling, attempting strike selection against a zero underlying price. This produces garbage results.

**Instances**: 16:31 UTC, 16:46 UTC, 17:11 UTC

**Fix**: If `spxPrice <= 0` or `spxPrice` is falsy, skip the cycle entirely and return early. Log a warning about bad data from the pipeline.

---

### 🟠 P1-5: Bracket Cancel Failure Not Handled Gracefully

**File**: `src/agent/position-manager.ts`, `cancelBracketLegs()` method  
**Status**: 🔧 FIXING  

**Problem**: When cancelling bracket/OCO legs before a sell, a 400 error is logged but execution continues. The subsequent sell then fails because the bracket legs still account for the position quantity.

**What happened**: At 12:16 ET, `Failed to cancel bracket #119403920 [6YA58635]: Request failed with status code 400`. The sell immediately after was rejected for qty mismatch.

**Fix**: 
- If bracket cancel fails, retry once after a short delay
- If still fails, query the order status — if already filled/cancelled/expired, proceed
- If bracket is still active, adjust sell quantity to exclude bracket-covered shares
- Consider: don't proceed with sell if bracket cancel fails; instead mark position for retry

### 🟠 P1-6: Expired Contract Validation Missing

**File**: `agent-xsp.ts`, `executeEntry()` / strike selection  
**Status**: 🔧 FIXING  

**Problem**: The agent attempted to trade `SPX260320P06440000` — a March 20 contract, **11 days expired**. The 1DTE logic also selected 0DTE contracts (`XSP260331`) overnight when it should have selected April 1.

**Fix**: Add strict expiry validation: refuse to trade any contract where expiry date ≤ today ET. Log and skip.

---

### 🟡 P2-7: Stale Quote Guard Missing

**File**: `agent-xsp.ts`, `executeEntry()` function  
**Status**: DOCUMENTED  

**Problem**: Overnight at ~4AM, the agent executed against stale quotes (options don't trade overnight). Quotes with `last` timestamps hours old were treated as valid.

**Fix**: Before entry, check if the quote's last trade is within a reasonable window (e.g., 5 minutes for RTH, reject overnight). This prevents phantom fills against stale data.

---

### 🟡 P2-8: Daily State Lost on PM2 Restart

**File**: `agent-xsp.ts`, global state variables  
**Status**: DOCUMENTED  

**Problem**: The agent was restarted 21 times on March 31 (hot-patches + crash loops). Each restart resets `dailyPnl`, `tradesTotal`, `winsTotal`, `consecutiveRejections` to zero. This means:
- P&L tracking is wrong (resets to $0 on each restart)
- Win rate is wrong
- Rejection backoff is defeated (counter resets)
- Daily trade limits can be exceeded

**Fix (future)**: Persist daily state to a file (`data/xsp-daily-state.json`) keyed by date. On startup, load existing state if the date matches. This survives PM2 restarts.

---

### 🟡 P2-9: Overnight Cycling (ALREADY FIXED)

**Status**: ✅ FIXED (commit `c855416`, 2026-03-31 08:15 UTC)  

The old code's `isMarketOpen()` was broken, causing the agent to cycle overnight for 12+ hours, burning 238 cycles that all hit "Past close cutoff". Fixed with proper ET time checks in the market hours function.

---

### 🟡 P2-10: Pre-Market Trading (ALREADY FIXED)

**Status**: ✅ FIXED (commit `c855416` + `9792e77`)  

Before the market hours fix, the agent executed 20 live orders between midnight and 4:15 AM ET against stale/phantom quotes. Fixed by the same market hours overhaul.
